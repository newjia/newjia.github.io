〇、序言

前面两篇文章多线程的概念和应用（【底层探索】- 多线程(一)、【底层探索】- 多线程三（GCD应用）），再加上平时经常在使用，下面来探索一下源码层面上的实现，加深对多线程的理解。

一、同步函数 (sync) 实现

1.1 代码展示

先看看同步函数，实现一个同步函数如下所示:

    dispatch_sync(dispatch_queue_create("doramon", NULL), ^{
        NSLog(@"ichi");
    });

会正常打印结果如下：

2020-05-19 01:31:41.178163+0800 ASYNC[19636:766754] ichi

1.2 源码探究

为了知道dispatch_sync的实现原理，直接在 libdispatch，里搜索关键字，结果如下：

得到dispatch_sync 的方法实现如下：

DISPATCH_NOINLINE
void
dispatch_sync(dispatch_queue_t dq, dispatch_block_t work)
{
    uintptr_t dc_flags = DC_FLAG_BLOCK;
    if (unlikely(_dispatch_block_has_private_data(work))) {
        return _dispatch_sync_block_with_privdata(dq, work, dc_flags);
    }
    _dispatch_sync_f(dq, work, _dispatch_Block_invoke(work), dc_flags);
}

实现代码里unlikely 这部分，意思是不太可能进入该分支，暂不考虑。

主要关心_dispatch_sync_f(dq, work, _dispatch_Block_invoke(work), dc_flags); 的实现，这里的block 回调参数work，被传入下一个实现，继续探寻，得到一个中转函数：

_dispatch_sync_f(dispatch_queue_t dq, void *ctxt, dispatch_function_t func,
        uintptr_t dc_flags)
{
    _dispatch_sync_f_inline(dq, ctxt, func, dc_flags);
}

得到实现函数入下:

DISPATCH_ALWAYS_INLINE
static inline void
_dispatch_sync_f_inline(dispatch_queue_t dq, void *ctxt,
        dispatch_function_t func, uintptr_t dc_flags)
{

    // 串行 来到这里
    //
    if (likely(dq->dq_width == 1)) {
        return _dispatch_barrier_sync_f(dq, ctxt, func, dc_flags);
    }

    if (unlikely(dx_metatype(dq) != _DISPATCH_LANE_TYPE)) {
        DISPATCH_CLIENT_CRASH(0, "Queue type doesn't support dispatch_sync");
    }

    dispatch_lane_t dl = upcast(dq)._dl;
    // Global concurrent queues and queues bound to non-dispatch threads
    // always fall into the slow case, see DISPATCH_ROOT_QUEUE_STATE_INIT_VALUE
    if (unlikely(!_dispatch_queue_try_reserve_sync_width(dl))) {
        return _dispatch_sync_f_slow(dl, ctxt, func, 0, dl, dc_flags);
    }

    if (unlikely(dq->do_targetq->do_targetq)) {
        return _dispatch_sync_recurse(dl, ctxt, func, dc_flags);
    }
    _dispatch_introspection_sync_begin(dl);
    _dispatch_sync_invoke_and_complete(dl, ctxt, func DISPATCH_TRACE_ARG(
            _dispatch_trace_item_sync_push_pop(dq, ctxt, func, dc_flags)));
}

这里关键函数有两个：

  • _dispatch_barrier_sync_f 函数。此时dq_width = 1，会进入栅栏函数，即队列宽度为1——同时只可以执行1个任务，也就是栅栏函数的原理。

        if (likely(dq->dq_width == 1)) {
        return _dispatch_barrier_sync_f(dq, ctxt, func, dc_flags);
    }

    继续探寻一下栅栏函数的内部实现，接下来的又是一个通用的函数：

    _dispatch_barrier_sync_f(dispatch_queue_t dq, void *ctxt,
        dispatch_function_t func, uintptr_t dc_flags)
{
    _dispatch_barrier_sync_f_inline(dq, ctxt, func, dc_flags);
}

    继续往下探索实现：

    dispatch_barrier_sync_f_inline(dispatch_queue_t dq, void *ctxt,
        dispatch_function_t func, uintptr_t dc_flags)
{
    dispatch_tid tid = _dispatch_tid_self();

    if (unlikely(dx_metatype(dq) != _DISPATCH_LANE_TYPE)) {
        DISPATCH_CLIENT_CRASH(0, "Queue type doesn't support dispatch_sync");
    }

    dispatch_lane_t dl = upcast(dq)._dl;
    // The more correct thing to do would be to merge the qos of the thread
    // that just acquired the barrier lock into the queue state.
    //
    // However this is too expensive for the fast path, so skip doing it.
    // The chosen tradeoff is that if an enqueue on a lower priority thread
    // contends with this fast path, this thread may receive a useless override.
    //
    // Global concurrent queues and queues bound to non-dispatch threads
    // always fall into the slow case, see DISPATCH_ROOT_QUEUE_STATE_INIT_VALUE
    if (unlikely(!_dispatch_queue_try_acquire_barrier_sync(dl, tid))) {
        return _dispatch_sync_f_slow(dl, ctxt, func, DC_FLAG_BARRIER, dl,
                DC_FLAG_BARRIER | dc_flags);
    }

    if (unlikely(dl->do_targetq->do_targetq)) {
        return _dispatch_sync_recurse(dl, ctxt, func,
                DC_FLAG_BARRIER | dc_flags);
    }
    _dispatch_introspection_sync_begin(dl);
    _dispatch_lane_barrier_sync_invoke_and_complete(dl, ctxt, func
            DISPATCH_TRACE_ARG(_dispatch_trace_item_sync_push_pop(
                    dq, ctxt, func, dc_flags | DC_FLAG_BARRIER)));
}

    这个才是完整的实现：

    1. _dispatch_tid_self：获取当前线程的ID，内部的实现是一个这样的宏：

      #define _dispatch_tid_self()        ((dispatch_tid)_dispatch_thread_port())

      接着看这个port 究竟是什么，还是一个宏。

      在32位环境下，只是将0 进行了语义转换：

      #if defined(_WIN32)
#define _dispatch_thread_port() ((mach_port_t)0)

      否则，在开启线程本地存储下，获取线程详细信息：

      #if DISPATCH_USE_DIRECT_TSD
#define _dispatch_thread_port() ((mach_port_t)(uintptr_t)\
        _dispatch_thread_getspecific(_PTHREAD_TSD_SLOT_MACH_THREAD_SELF))

      未开启直接本地线程存储情况下，是这样的实现:

      #define _dispatch_thread_port() pthread_mach_thread_np(_dispatch_thread_self())

      继续追踪，刨根问底儿，看看getspecific 究竟是实现了什么：

      #define _dispatch_thread_getspecific(key) \
    (_dispatch_get_tsd_base()->key)
#define _dispatch_thread_setspecific(key, value) \
    (void)(_dispatch_get_tsd_base()->key = (value))

      get 的时候，看到了key；set的时候，可以观察到是按照KEY-VALUE 的格式存储到内存中。好的，这个部分 先暂时到这里，知道了获取tid 的流程。

    2. _dispatch_sync_f_slow 函数：

      这个分支的函数实现是：

          if (unlikely(!_dispatch_queue_try_acquire_barrier_sync(dl, tid))) {
        return _dispatch_sync_f_slow(dl, ctxt, func, DC_FLAG_BARRIER, dl,
                DC_FLAG_BARRIER | dc_flags);
    }

      从unlikely来看，是尝试获取栅栏+同步的实现。内部实现看看：

      _dispatch_sync_f_slow(dispatch_queue_class_t top_dqu, void *ctxt,
        dispatch_function_t func, uintptr_t top_dc_flags,
        dispatch_queue_class_t dqu, uintptr_t dc_flags)
{
        struct dispatch_sync_context_s dsc = {
        .dc_flags    = DC_FLAG_SYNC_WAITER | dc_flags,
        .dc_func     = _dispatch_async_and_wait_invoke,
        .dc_ctxt     = &dsc,
        .dc_other    = top_dq,
        .dc_priority = pp | _PTHREAD_PRIORITY_ENFORCE_FLAG,
        .dc_voucher  = _voucher_get(),
        .dsc_func    = func,
        .dsc_ctxt    = ctxt,
        .dsc_waiter  = _dispatch_tid_self(),
    };
    _dispatch_trace_item_push(top_dq, &dsc);
    __DISPATCH_WAIT_FOR_QUEUE__(&dsc, dq);
}

      • 队列里面，将结构体类型的上下文进行压栈。

      • 接下来是重点，接下来有个__DISPATCH_WAIT_FOR_QUEUE__ 的宏，看看实现如下：

        __DISPATCH_WAIT_FOR_QUEUE__(dispatch_sync_context_t dsc, dispatch_queue_t dq)
{
    uint64_t dq_state = _dispatch_wait_prepare(dq);
    if (unlikely(_dq_state_drain_locked_by(dq_state, dsc->dsc_waiter))) {
        DISPATCH_CLIENT_CRASH((uintptr_t)dq_state,
                "dispatch_sync called on queue "
                "already owned by current thread");
    }

        这里的dq_state 状态是队列中等待的状态，而 unlikely 判断的是_dq_state_drain_locked_by(dq_state, dsc->dsc_waiter)，即当前线程的上下文dsc与队列等待比较。

        具体实现如何：

        _dq_state_drain_locked_by(uint64_t dq_state, dispatch_tid tid)
{
    return _dispatch_lock_is_locked_by((dispatch_lock)dq_state, tid);
}
// 实现如下：
_dispatch_lock_is_locked_by(dispatch_lock lock_value, dispatch_tid tid)
{
    // equivalent to _dispatch_lock_owner(lock_value) == tid
    return ((lock_value ^ tid) & DLOCK_OWNER_MASK) == 0;
}

        可以看到，是将lock_value（队列等待状态） 与 tid（当前线程ID）做异或判断，如果两者不同，返回TRUE 。而DLOCK_OWNER_MASK的实现是一个确定的16进制数，可以不用考虑 

        #define DLOCK_OWNER_MASK            ((dispatch_lock)0xfffffffc)

        回到上一个函数再继续看，当_dq_state_drain_locked_by 函数返回结果为1时，会触发CRASH，并报错如下：

                DISPATCH_CLIENT_CRASH((uintptr_t)dq_state,
                "dispatch_sync called on queue "
                "already owned by current thread");

        同步函数调用（失败），已被当前线程持有

        这就是线程死锁的由来

1.3 小结

  1. 同步函数的创建，是建立在资源的独占性，即同时只可以执行一个任务。
  2. 当队列宽度等于1，此时任务依次执行
  3. 当队列等待与当前线程任务不一致时，会导致线程死锁，即同步函数调用一个被当前线程占用的资源（这是不允许的，互斥性的），会导致崩溃的出现。
